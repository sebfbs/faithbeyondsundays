### Pipeline Reliability — BUILT + DEPLOYED ✅
**Problem:** `upload-sermon` triggered `process-sermon` via a fire-and-forget HTTP call. If it failed, the job sat queued forever.

**Solution: Belt and suspenders**
- Created `supabase/functions/sermon-job-poller/index.ts` — thin wrapper that calls `process-sermon` on a cron schedule
- Removed fire-and-forget from `upload-sermon/index.ts`
- Supabase DB webhook configured: fires on `sermon_jobs` INSERT → instantly calls `process-sermon`
- pg_cron schedule set: calls `sermon-job-poller` every 1 minute as safety net
- All three functions deployed

### Chapter Timestamp Grounding — BUILT + DEPLOYED ✅
**Problem:** AI was generating chapter timestamps that could be hallucinated or out of range.

**Solution:**
- After ElevenLabs transcription, extract list of real timestamps embedded in transcript
- Pass that list explicitly to the chapter generation prompt: "only use timestamps from this list"
- After AI returns chapters, validate each timestamp — if not in the valid list, substitute the nearest real one
- Applied to both initial generation and admin regeneration paths

## What's Ready to Build Next Session

### 1. Async ElevenLabs Transcription (HIGHEST PRIORITY)
**Why:** 60-minute sermon videos will hit Supabase's 150-second Edge Function timeout during synchronous transcription. Must fix before real-world testing.

**Plan (3 files, fully designed):**

**NEW: `supabase/functions/elevenlabs-webhook/index.ts`**
- Receives completed transcript POST from ElevenLabs when async processing finishes
- Verifies request via secret in webhook URL query param
- Saves transcript + word timings to `sermon_transcripts`
- Handles both "completed" and "failed" ElevenLabs statuses
- If completed: resets job to "queued", clears lock and request_id → poller picks up within 60 seconds → AI generation runs
- If failed: marks job as retrying or failed
- Returns 200 immediately (or ElevenLabs retries)
- If payload parsing fails: return 200, mark job failed, log raw payload to error_details for debugging

**MODIFIED: `supabase/functions/process-sermon/index.ts`**
- Transcription section: send async request with `webhook: true`, get back `request_id`, save to `error_details`, set lease to 2 hours, return immediately
- If job already has `request_id` in `error_details` and lease is still valid: just refresh lease and return (don't re-send)
- Stale recovery section: also clear `error_details` when recovering a stale job — so if webhook never arrived, next run re-sends to ElevenLabs fresh
- Signed URL expiry: increase from 1 hour to 2 hours for async mode
- Everything else unchanged

**No DB migration needed** — `request_id` stored in existing `error_details JSONB` column

**Deploy:**
```bash
npx supabase functions deploy elevenlabs-webhook
npx supabase functions deploy process-sermon
```

**One config step (Sebastian does once in ElevenLabs dashboard):**
Set webhook URL to:
`https://[project-ref].supabase.co/functions/v1/elevenlabs-webhook?secret=YOUR_SECRET`
Add `YOUR_SECRET` as an Edge Function env var: `ELEVENLABS_WEBHOOK_SECRET`

**Bugs addressed in design:**
- Webhook never arrives → 2-hour lease → stale recovery clears request_id → re-sends fresh
- ElevenLabs reports failure → job marked failed/retrying correctly
- Signed URL expiry → bumped to 2 hours
- Unknown webhook payload format → defensive parsing, raw payload logged on failure
